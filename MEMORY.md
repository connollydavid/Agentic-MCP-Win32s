## 2026-06-06 — Repo separation

- The original inline MCP-Win32s repo was split: agentic assets (plans, harness config, process docs) moved to this host repo; the software itself is vendored as the `mcp-win32s/` submodule. PLAN.md was split into `plan/PHASE1-7.md` with strict sequencing rules in `plan/PLAN.md`.
- **Submodule paths are lowercase** (`mcp-win32s/`, `andrej-karpathy-skills/`). A WSL2 9p/drvfs bug poisoned the dentry for the mixed-case path `MCP-Win32s` after a rename-then-failed-clone sequence; the poison survived `drop_caches` and remounts. Lowercase paths sidestep the entire class of case-mismatch problems on /mnt/c — prefer them for all new top-level entries.
- The allium plugin is registered project-scope in `~/.claude/plugins/installed_plugins.json` keyed by **absolute project path** — moving/renaming the project directory silently orphans the plugin (`/reload-plugins` reports 0 plugins). `.claude/settings.json` here carries `extraKnownMarketplaces` + `enabledPlugins` so a fresh clone self-registers.
- Planning state must exist in exactly one place: `plan/PLAN.md` index here. The submodule's CLAUDE.md previously carried a duplicate phase-status table (removed in MCP-Win32s@91df55a) and host AGENTS.md was a stale copy of the submodule's constraints (rewritten as a source-of-truth map). Watch for this drift pattern when adding docs.
- `mcp-win32s/specs/transport.allium` has 2 pre-existing parse errors at line 202: a stray `@` and an `invariant SerialHasNoOrderlyClose` with a comment but no body (truncated edit, committed). The merge gate ("allium check clean") was therefore not actually met at MCP-Win32s@28055d7. Fix must go through `/allium:tend` once the plugin session loads.
- Version drift found 2026-06-06: allium CLI 3.2.3 installed vs 3.2.4 on crates.io; allium plugin 3.1.5 cached vs 3.3.0 in the juxt-plugins marketplace (3.3.0 adds tend/weed agents). Update via `cargo install allium-cli` and `/plugin update allium@juxt-plugins`.
- The `@invariant` prose form is only valid **inside contract declarations** (allium 3); top-level prose properties go in `--` comments, expression invariants use `invariant Name { ... }`. The transport.allium parse errors (see 2026-06-06 entry above) were exactly this misuse — fixed via tend in MCP-Win32s@e143fef; `allium check` now reports info-only across all three specs.

## 2026-06-06 — Phase 4 implement lessons
- theft/UBSan host harness found real UB on a client-controlled path: `Base64Decode`'s rolling `(buf << 6)` overflowed a signed int. Fixed (unsigned accumulator). Lesson: the host-native sanitizer pass catches what the C89/Wine suite cannot.
- Windows job-object semantics correction: `JOB_OBJECT_LIMIT_PROCESS_MEMORY` does NOT kill the child — it fails allocations; only `JOB_OBJECT_LIMIT_PROCESS_TIME` terminates (quota exit 1816). PHASE4's "memory cap kills" assumption was wrong; killed_by=memory_cap is only attributable via the quota exit code.
- Spawned hidden children inherit the parent's error mode: without `SetErrorMode(SEM_FAILCRITICALERRORS|SEM_NOGPFAULTERRORBOX|SEM_NOOPENFILEERRORBOX)` around `CreateProcessA`, a child that fails to start (e.g. memory-capped: loader box 0xC000012D) pops a BLOCKING dialog on the server's desktop and hangs the capture loop. This bit the dev desktop repeatedly during testing.
- Blocked anonymous-pipe `ReadFile` reader threads cannot be cancelled portably (closing the read end does not unblock them). Children that may be left running (VDM orphans) must be captured with the non-blocking PeekNamedPipe polling loop even on threaded hosts.
- Under WSL interop, a ConPTY child's stdout routes to the inherited real console, not the pseudoconsole pipe — PTY echo content assertions must skip-with-reason locally; they assert fully on native Windows/CI.

## 2026-06-06 — Phase 4 review gate caught a catalog-gate bypass (3rd time the gate paid off)
- The shell-builtin auto-route in HandleExec applied ArgvCmdEscape (Q15 caret-escape) ONLY to the external route; the builtin route copied the catalog shell string + ArgvJoin(args) verbatim. A no-whitespace cmd metacharacter in a positional (argv:["dir","x&calc"]) is emitted bare by ArgvJoin and the shell treats & as a separator → ran uncatalogued "calc" past the enforced catalog with no unsafe flag. The full Allium lifecycle (check/propagate/weed) and CI all passed it; only the fresh-context adversarial sub-agent found it. Root cause the spec hid it: catalog.allium abstracted the command line into effective_cmd_line(cmd), so the escaping divergence wasn't pinnable. Fix added a ShellTailNeutralised security invariant naming that both shell routes must escape identically.
- Lesson: when a spec uses a black-box function to abstract a security-relevant transformation (escaping, sanitisation), the abstraction can hide a divergence the weed audit cannot see. Pin the security property explicitly (invariant) rather than burying it in an opaque helper.

## 2026-06-07 — Phase 5.0 bridge first-green + a phase-gate arming gotcha
- Phase 5.0 (bridge core, Rust on rmcp 1.7) reached first-green: `cargo test` 8 green (5 integration incl. a full MCP lifecycle over an in-memory duplex + 3 proptest) + MCP Inspector CLI conformance (tools/list shows win32_echo with a JSON-Schema-2020-12 inputSchema; tools/call → {content, structuredContent, isError:false}). The deterministic gate was extended with Gate 7 (bridge `cargo test`) and `bridge/{src,tests,examples,Cargo.toml}` folded into the green-hash.
- Operational gotcha (cost me a near-collision): the `/phase-gate` Stop hook runs ACROSS turns once armed, and a long verification command auto-backgrounds. If you arm the gate AND kick off a heavy verification in the same step, ending the turn fires the Stop hook as a SECOND concurrent gate run — two `ninja` builds in the same `build/mingw`, two wire-smoke servers on TCP 31798. Safe sequence: run the full gate suite (e.g. by invoking the hook once) to populate `.claude/.phase-gate.greenhash` while DISARMED, confirm the recomputed change-hash equals the stored green-hash, THEN write the marker to arm. With a matching green-hash the next turn-end Stop does allium-check + hash-match and exits instantly — no second heavy run.
- The green-hash set deliberately excludes `bridge/Cargo.lock` and `bridge/target/`, so a `cargo test` run (which may touch Cargo.lock) does not invalidate the hash.

## 2026-06-07 — Phase 5.0 weed: a spec can be confidently wrong about a delegated library
- The bridge delegates the MCP lifecycle + version negotiation to rmcp. The spec's `VersionNegotiated` asserted "client version ≠ ours → negotiate to ours (2025-11-25)". rmcp actually DOWN-negotiates: a client requesting an older *supported* revision (e.g. 2025-06-18) gets its OWN version echoed back, not clamped up (verified in rmcp-1.7.0/src/service/server.rs:230-238 — `partial_cmp(client, server)`; Less ⇒ echo client). The spec's else-branch was a real faithfulness defect, not an abstraction gap. `allium check`, propagate, and a green build all passed it; only the adversarial weed auditor that READ THE LIBRARY SOURCE caught it. Lesson: when a spec models a delegated dependency's behaviour as fact, the weed audit must verify against the dependency's actual source, not its prose/our assumption — the plausible-but-wrong claim is exactly the failure mode. Fix: spec corrected to `client<=bridge?client:bridge` + a `NegotiatedVersionIsCommonFloor` invariant pinning the delegated transform, and an integration test (`version_negotiation_is_down_to_the_common_floor`) that drives an older client and asserts the echoed version via the server's `peer_info` — so a future rmcp bump that changes negotiation is caught.
- Also: the capability gate's prune enforcement (`tools_to_prune`→`remove_route`→absence from `tools/list`) is UNVERIFIED end-to-end in 5.0 because `GATED_TOOLS` is empty — recorded gap G1, a hard precondition on 5.3 (the first work-item that gates a real tool, `win32_peek`/`poke`). The `satisfies()` decision is fail-closed under every adversarial input (no bypass), but "a gated tool is actually hidden" has no executable witness until a gated tool exists.

## 2026-06-07 — Phase 5.0 bridge merged (PR #11); review gate caught a missing-CI false-green
- 5.0 (Rust MCP bridge on rmcp) merged to mcp-win32s main as #11 (squash 5a3d0a5); host pointer bumped (9f5df63). Full lifecycle ran: tend → propagate → implement (first-green) → weed → review → observed CI.
- Lesson (generalizable): when a NEW crate/language lands beside an existing one (here Rust bridge/ beside the C project), it needs its OWN CI job, or "observed CI" is a false green. The existing workflow built/tested only the C project via MinGW+Wine, so the entire +1279-line bridge would have merged with NO automated gate — `cargo test` green was a local-only proxy. The weed audit and every tool missed this (they check spec↔code, not CI coverage); only the adversarial review gate caught it. Fix: added a `bridge` job (cargo fmt --check / clippy -D warnings / build --release / test) to .github/workflows/build-and-test.yml, and committed bridge/Cargo.lock (a binary crate must pin its lockfile, else a dependency re-resolve can drift the rmcp behaviour the negotiation witness test pins).
