2026-06-06 — Repo separation

  • The original inline MCP-Win32s repo was split: agentic assets (plans, harness config, process docs) moved to this host repo; the software itself is vendored as the mcp-win32s/ submodule. PLAN.md was split into plan/PHASE1-7.md with strict sequencing rules in plan/PLAN.md.
  • Submodule paths are lowercase (mcp-win32s/, andrej-karpathy-skills/). A WSL2 9p/drvfs bug poisoned the dentry for the mixed-case path MCP-Win32s after a rename-then-failed-clone sequence; the poison survived drop_caches and remounts. Lowercase paths sidestep the entire class of case-mismatch problems on /mnt/c — prefer them for all new top-level entries.
  • The allium plugin is registered project-scope in ~/.claude/plugins/installed_plugins.json keyed by absolute project path — moving/renaming the project directory silently orphans the plugin (/reload-plugins reports 0 plugins). .claude/settings.json here carries extraKnownMarketplaces + enabledPlugins so a fresh clone self-registers.
  • Planning state must exist in exactly one place: plan/PLAN.md index here. The submodule's CLAUDE.md previously carried a duplicate phase-status table (removed in MCP-Win32s@91df55a) and host AGENTS.md was a stale copy of the submodule's constraints (rewritten as a source-of-truth map). Watch for this drift pattern when adding docs.
  • mcp-win32s/specs/transport.allium has 2 pre-existing parse errors at line 202: a stray @ and an invariant SerialHasNoOrderlyClose with a comment but no body (truncated edit, committed). The merge gate ("allium check clean") was therefore not actually met at MCP-Win32s@28055d7. Fix must go through /allium:tend once the plugin session loads.
  • Version drift found 2026-06-06: allium CLI 3.2.3 installed vs 3.2.4 on crates.io; allium plugin 3.1.5 cached vs 3.3.0 in the juxt-plugins marketplace (3.3.0 adds tend/weed agents). Update via cargo install allium-cli and /plugin update allium@juxt-plugins.
  • The @invariant prose form is only valid inside contract declarations (allium 3); top-level prose properties go in -- comments, expression invariants use invariant Name { ... }. The transport.allium parse errors (see 2026-06-06 entry above) were exactly this misuse — fixed via tend in MCP-Win32s@e143fef; allium check now reports info-only across all three specs.

2026-06-06 — Phase 4 implement lessons

  • theft/UBSan host harness found real UB on a client-controlled path: Base64Decode's rolling (buf << 6) overflowed a signed int. Fixed (unsigned accumulator). Lesson: the host-native sanitizer pass catches what the C89/Wine suite cannot.
  • Windows job-object semantics correction: JOB_OBJECT_LIMIT_PROCESS_MEMORY does NOT kill the child — it fails allocations; only JOB_OBJECT_LIMIT_PROCESS_TIME terminates (quota exit 1816). PHASE4's "memory cap kills" assumption was wrong; killed_by=memory_cap is only attributable via the quota exit code.
  • Spawned hidden children inherit the parent's error mode: without SetErrorMode(SEM_FAILCRITICALERRORS|SEM_NOGPFAULTERRORBOX|SEM_NOOPENFILEERRORBOX) around CreateProcessA, a child that fails to start (e.g. memory-capped: loader box 0xC000012D) pops a BLOCKING dialog on the server's desktop and hangs the capture loop. This bit the dev desktop repeatedly during testing.
  • Blocked anonymous-pipe ReadFile reader threads cannot be cancelled portably (closing the read end does not unblock them). Children that may be left running (VDM orphans) must be captured with the non-blocking PeekNamedPipe polling loop even on threaded hosts.
  • Under WSL interop, a ConPTY child's stdout routes to the inherited real console, not the pseudoconsole pipe — PTY echo content assertions must skip-with-reason locally; they assert fully on native Windows/CI.

2026-06-06 — Phase 4 review gate caught a catalog-gate bypass (3rd time the gate paid off)

  • The shell-builtin auto-route in HandleExec applied ArgvCmdEscape (Q15 caret-escape) ONLY to the external route; the builtin route copied the catalog shell string + ArgvJoin(args) verbatim. A no-whitespace cmd metacharacter in a positional (argv:["dir","x&calc"]) is emitted bare by ArgvJoin and the shell treats & as a separator → ran uncatalogued "calc" past the enforced catalog with no unsafe flag. The full Allium lifecycle (check/propagate/weed) and CI all passed it; only the fresh-context adversarial sub-agent found it. Root cause the spec hid it: catalog.allium abstracted the command line into effective_cmd_line(cmd), so the escaping divergence wasn't pinnable. Fix added a ShellTailNeutralised security invariant naming that both shell routes must escape identically.
  • Lesson: when a spec uses a black-box function to abstract a security-relevant transformation (escaping, sanitisation), the abstraction can hide a divergence the weed audit cannot see. Pin the security property explicitly (invariant) rather than burying it in an opaque helper.

2026-06-07 — Phase 5.0 weed: a spec can be confidently wrong about a delegated library

  • The bridge delegates the MCP lifecycle + version negotiation to rmcp. The spec's VersionNegotiated asserted "client version ≠ ours → negotiate to ours (2025-11-25)". rmcp actually DOWN-negotiates: a client requesting an older supported revision (e.g. 2025-06-18) gets its OWN version echoed back, not clamped up (verified in rmcp-1.7.0/src/service/server.rs:230-238 — partial_cmp(client, server); Less ⇒ echo client). The spec's else-branch was a real faithfulness defect, not an abstraction gap. allium check, propagate, and a green build all passed it; only the adversarial weed auditor that READ THE LIBRARY SOURCE caught it. Lesson: when a spec models a delegated dependency's behaviour as fact, the weed audit must verify against the dependency's actual source, not its prose/our assumption — the plausible-but-wrong claim is exactly the failure mode. Fix: spec corrected to client<=bridge?client:bridge + a NegotiatedVersionIsCommonFloor invariant pinning the delegated transform, and an integration test (version_negotiation_is_down_to_the_common_floor) that drives an older client and asserts the echoed version via the server's peer_info — so a future rmcp bump that changes negotiation is caught.
  • Also: the capability gate's prune enforcement (tools_to_prune→remove_route→absence from tools/list) is UNVERIFIED end-to-end in 5.0 because GATED_TOOLS is empty — recorded gap G1, a hard precondition on 5.3 (the first work-item that gates a real tool, win32_peek/poke). The satisfies() decision is fail-closed under every adversarial input (no bypass), but "a gated tool is actually hidden" has no executable witness until a gated tool exists.

2026-06-07 — Phase 5.1 merged (PR #12); API-first file ops, clean run on a fully-worked lifecycle

  • 5.1 (copy/move/mkdir/rmdir device commands + the eight win32_* bridge file tools) merged to mcp-win32s main as #12 (squash fce5ce8); host pointer bumped (b68a5ff). Full lifecycle: tend → propagate (83 obligations) → implement (fan-out: frozen interfaces first, 3 parallel module agents on disjoint files, integration seams in the main session, independent re-verify) → distill (no-op) → weed (2 parallel adversarial auditors) → review → observed CI.   • Design decisions settled in-stage with the user, now pinned in spec + test (not just prose): (1) copy = CopyFileA bFailIfExists=TRUE — a copy can NEVER clobber; overwrite is an explicit delete-then-copy. This is WHY win32_copy_file honestly carries no destructiveHint (the D6 hint table stays honest only because the device physically can't overwrite). (2) mkdir = single CreateDirectoryA, no mkdir -p (missing parent → "path not found", the agent composes). (3) rmdir relies on RemoveDirectoryA's own non-empty refusal, no recursive sweep. (4) bridge relay maps MCP arg source→wire "path", dest→wire "dest" — the arg→wire mapping is itself a safety-relevant transform, pinned by FileToolsAreDirectRelays + an observed-wire test with an explicit no-"source"-field negative assertion.
  • Lesson (safety-pin test quality): a destruction-guard test must assert the PROTECTED DATA IS UNTOUCHED, not merely that an error returned. copy_dest_exists_errors reads the dest and byte-compares it unchanged; rmdir_nonempty_errors reads the contained file back; mkdir_no_recursive_create asserts the deep path was NOT created. An "error returned" assertion would pass a FALSE-bFailIfExists implementation that overwrote and then errored on something else. The weed auditor and the review reviewer both explicitly checked "would this test fail on the bypassed code?" — that question, not coverage count, is the pin.
  • Lesson (anti-slop boundary in practice): the user caught a numbered work-item reference ("5.1 additions") leaking into a submodule spec COMMENT mid-tend. The sanctioned Phase N/work-item vocabulary lives ONLY in the host plan/ — in mcp-win32s specs/src/tests/commits it is slop. Scrub it from comments and commit subjects as you write, not just from code.

2026-06-08 — Phase 5.4 merged (PR #15); device-owned full-range text encoding

  • 5.4 (the project's largest single work-item: the device owns the ENTIRE text pipeline and emits valid UTF-8 across Win3.1+Win32s→Win11 using our OWN embedded C89 charset tables, never the OS's MultiByteToWideChar/WideCharToMultiByte — a hand-rolled UTF-8↔UTF-16 codec @50k theft trials; 30 generated codepages (SBCS + four CJK DBCS) from cited Unicode Consortium data + a committed generator + a non-bijection matrix; three OS tiers manifest/wide/codepage; the nine delay-loaded -W APIs; CreateProcessW-on-NT spawn; the file/exec/listing seam; the bridge collapsed to a UTF-8 passthrough) merged to mcp-win32s main as #15 (squash 470ded9); host pointer bumped (36f3cb3). Full lifecycle tend→propagate(15 structural + 7 safety pins)→implement(M1–M6 fan-out + the main-session seam)→distill(no-op, spec-first)→weed(4 parallel adversarial auditors)→review(APPROVE, zero findings)→observed CI green (both jobs). Seven named safety pins: NeverEmitInvalidUtf8, StrictNarrowingRejectsUnrepresentable, LossyDecodeIsTotal, TruncationIsBoundaryClean, CodepageRoundTripsOnBijectiveSubset, PathSeparatorScanIsDbcsSafe, Utf8RoundTripFidelity.
  • Lesson (CI parity has a COMPILER-VERSION axis, not just Wine-vs-native runtime): a full local ctest was green yet CI had been RED since the M1 push — CI's ubuntu-latest mingw GCC (12/13) is NEWER than the dev host's mingw GCC 10, and its -Wmaybe-uninitialized under -Werror rejected the codec PBT input buffers, which are filled in a loop to a RUNTIME-variable length (PROP_INT(0, MAX_B), may be 0) then passed to the codec — newer GCC can't prove full init across the runtime-bounded loop; GCC 10 accepted it silently. The prior session pushed without watching the run, so the failure was latent for several commits. Reproduced with the dev host's NATIVE gcc 11 (the warning is GCC-version-dependent, not target-dependent — compile the host-pure codec path at -O2 -Werror=maybe-uninitialized), fixed by zero-initialising at declaration (trivially provable for any GCC). Always WATCH the actual CI run (gh run watch) before claiming the merge gate's parity tier; local green proves neither the Wine runtime nor the CI compiler version.
  • Lesson (the orchestrator must REFUTE an auditor BLOCKER with the COMPILER, not the narrative — C89 block scope edition): a weed auditor raised a BLOCKER that encoding.c had "C89 mid-block declarations" (variables declared inside loop/if bodies). FALSE: C89 permits declarations at the TOP of ANY nested block, including a loop/if body, and the violation is only a declaration AFTER a statement in the SAME block (mixed declarations and code). The auditor conflated the two. Definitive refutation: the file compiles clean under -std=c89 -pedantic -pedantic-errors -Werror -Wdeclaration-after-statement (the exact diagnostic for a real mixed-declaration violation) — a machine proof beats the narrative. Auditor reports are never trusted; a deterministic claim gets a deterministic check.
  • Lesson (the device owns the whole text pipeline; the DBCS-safe self-synchronising property is what makes the seam safe): all transcoding lives on the device (UTF-8 on every tier via our own tables), so the bridge becomes a near-passthrough — the encoding ready tag is informational provenance that gates NO tool (the old "utf8" capability was retired because it was dead). The seam's two safety properties both rest on UTF-8 being self-synchronising: (1) a path-separator scan on the UTF-8 form before the single narrowing point can't mis-split a DBCS pair (a 0x80–0xBF continuation byte is never 0x5C); (2) the CreateProcessW widen can't bypass the catalog gate because the gate runs on the UTF-8 argv BEFORE assembly+widen AND a ≥0x80 UTF-8 byte can never widen to an ASCII (<0x80) shell metacharacter — the same principle applied to argv. File READ/WRITE DATA is never transcoded (only paths/listings/exec-output are text).

2026-06-11 — Phase 6.2: Win32s serial is NOT a wall — direct ring-3 UART I/O works bidirectionally (Finding #3 root-caused; operator-consented)

  • Root cause of the 0-bytes serial (Finding #3): NOT a Win32s limitation and NOT the harness — it is the stubbed Win32 communications API. Win32s 1.25a exports SetCommState/SetCommTimeouts (so the DLL- and function-level allowlists pass them) but they are stubs returning err 120 ERROR_CALL_NOT_IMPLEMENTED; with that tolerated, ReadFile/WriteFile on the COM handle move 0 bytes. The Win32 serial layer is simply absent on this tier. (Corrects the line 89 "possible Win32s serial limitation" guess: the handle/comm path is dead, but the hardware is fully reachable — see below.)
  • The fix is to bypass the comm stack entirely and drive the 16550 UART by DIRECT ring-3 port I/O (IN/OUT to 0x3F8–0x3FF). Proven live on the pinned guest with a throwaway CRT-free probe (tools/phase6-qemu/uart-probe.c → UARTPROB.EXE, built -nostdlib -e _probemain, imports only {kernel32,user32} so it loads on Win32s): (1) the bare ring-3 IN/OUT is GRANTED — no #GP (scratch reg AA/55 round-tripped; the app did not fault) — same mechanism the in-VM COMM.DRV itself uses, and VCD passes System-VM port access through when nothing else owns the port; (2) TX works — host nc on 127.0.0.1:31800 captured UARTPROBE-TX\r\n straight off COM1, no comm stack / no COMM.DRV in the path; (3) RX works — the guest polled LSR and read 17 bytes HELLO-FROM-HOST\r\n sent from the host. Full duplex.
  • Why COMM.DRV is a non-issue (the operator's "disable COMM.DRV" reframed): COMM.DRV is passive until something calls OpenComm/CreateFile on the port (only then does it program the divisor, hook IRQ4, run its ISR). The direct design never opens COM1, so COMM.DRV never initialises that UART — there is nothing to "disable" and nothing to contend with. The only gatekeepers were the System-VM I/O privilege (granted, proven) and VCD (passes through, proven). No SYSTEM.INI edit was needed.
  • Architectural implication: direct-UART (option D) supersedes deferred-option-A (the 16-bit Universal Thunk through COMM.DRV) — it is cleaner and keeps the single-binary / zero-dependency / C89 constraints (A needed a Win16-toolchain companion DLL). Not yet device code; an architectural decision (implement a Win32s direct-UART transport backend now vs. record-and-defer + merge the loadability fixes first) is pending.
  • Security boundary (operator expressly consented — David, 2026-06-11): permissible because it is a bare ring-3 OUT with NO escalation — no ring-0, no VxD, no IOPL flip, no call gate, no IRQ hook (the design is polled, with exclusive ownership — never OpenComm). That is "use what the OS already grants user mode," not "manufacture access by escalating" — squarely inside the project posture. When this becomes device code it MUST carry a backing invariant pinning "bare ring-3 port I/O only; no VxD/IOPL/call-gate; UART owned exclusively (COM1 never opened via the OS)", and the weed gate-bypass dimension audits it. Consent + the boundary line are the MCP05/least-privilege "bounded, consented, logged" posture in practice.
  • Lesson (generalisable): on a thunked/subset Win32, an exported entry point can be a stub — presence in the export table (even at function granularity) does not mean it is implemented. SetCommState is exported AND allowlisted AND Wine-backed (so CI was green) yet returns err 120 on real Win32s. Probe behaviour, not just the symbol table. And when an OS subset stubs a high-level API, the lower-level primitive it was wrapping may still be fully available to ring-3 — here the entire Win32 comm layer is dead but the UART hardware underneath is directly reachable.
